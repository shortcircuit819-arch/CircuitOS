## What 0.7 is

CircuitOS moves from a single-user local Windows app to a multi-tenant, cloud-backed
platform where streamers can log in with Twitch while keeping local mode as the default. Cloud mode is optional for users who want hosted/profile-backed behavior; Twitch capabilities are not cloud-only. The 0.5 `IDataStore` seam made this a
*swap*, not a rewrite.

## Landed this development cycle

### The app runs on the cloud (`--cloud`)
- **Appwrite data layer** (`AppwriteDataStore`, model A): one row per
  `(userId, profileId, dataKey)` in the `profile_data` table, JSON blob in the `json`
  column. The admin panel reads/writes catalog, profile, boost, and inventory from
  Appwrite — same UI, cloud data. Saves write back; verified live.
- **Host-agnostic service:** `CircuitService` now takes `IDataStore` (not the local
  file store), with the filesystem-bound bits (Streamer.bot path, overlay) degrading
  gracefully so the same app drives either backend.
- **Cloud profile management** — list / create / rename / switch / delete, stored inside
  `profile_data`. Verified live.
- **Cloud backup recovery point** — every cloud save snapshots the prior version (under a
  `#bak` namespace); the Backups view can restore it. Verified live.

### Twitch authentication (Phase 3)
- **Direct Twitch OAuth** (no Auth0 in the desktop bridge). The streamer registers their
  own Twitch app; login uses a desktop authorization-code flow with a loopback redirect.
- **In-app login/logout** from the sidebar session panel, plus a `--twitch-login` CLI.
  The authenticated **Twitch user id becomes the cloud tenant** (replacing `local-dev`).
- **Session UI** — the footer badge opens a panel showing the Twitch identity, user id,
  token expiry, backend (cloud/local), and data location, with Log in / Log out controls.
  Tokens are cached locally only (gitignored); logout clears them.

### Native Twitch bridge (Phase 4, desktop)

- **Overview Pull Rates editing was tightened** — the Pull Rates panel no longer uses whole-card navigation, the Tune in Rate Lab button still navigates, sliders can be dragged without jumping views, and slider fill now uses the raw range percentage so the colored bar does not overshoot past the handle.
- **Sidebar and topbar theme coverage** — Appearance colors now drive sidebar/topbar chrome variables so custom themes reach the navigation shell, not only content panels.
- **Attach-only Twitch rewards now record without fulfillment errors** — attached rewards not created by this Twitch app skip fulfillment/cancel calls, avoiding Twitch 403 errors while still recording pulls and sending chat.
- **Native multi-profile routing was hardened** — live profiles now require unique command words and unique redemption titles; duplicate Twitch reward ids are blocked; the native listener refreshes after login/logout, profile live-state changes, and reward sync/edit/delete so reward-id routing does not go stale. Live Twitch verification with two rewards passed; attach-only rewards now skip fulfillment/cancel while still recording and sending chat.
- **Twitch Settings page** — the admin sidebar now has a dedicated Twitch page with account login/logout, simple workflow cards, and live-profile reward management. The page can load current Twitch channel-point rewards into a dropdown, attach an existing reward to a live profile, or create/sync from the profile redemption name. Reward ids persist per profile and the listener reuses stored mappings on restart. Edit updates managed reward title/cost through Helix and keeps the profile redemption name in sync. Delete removes synced CircuitOS-managed rewards and clears the local mapping after confirmation; attached non-manageable rewards are marked attach-only and are not edited or deleted by CircuitOS. The page also includes permissions/re-login guidance for reward management, redemption intake, and chat replies.
- **Reward creation/update** via Twitch Helix has been added as a diagnostic path.
- **EventSub WebSocket redemptions** have been live-verified in the desktop bridge: a real
  channel-point redemption routed to the right profile, ran the shared redemption pipeline,
  saved inventory, and fulfilled the redemption.
- **Pull announcements in Twitch chat** are now formatted centrally and sent after fulfillment.
- **Native chat commands** are wired through the shared command engine. Existing tokens may need
  one re-login to consent to the newer chat scopes.
- **The running app auto-starts the Twitch listener** when Twitch is configured and profiles are
  live, so the desktop bridge no longer requires a separate listener process.

### Foundations
- **`PullEngine`** — the dup-protection → tier-weighted → variant roll logic extracted
  from the Streamer.bot action into shared, tested .NET code (the single source of truth
  the Phase-4 native EventSub path will call). Distribution-tested (70/25/5 tiers, etc.).
  Variant labels are now trimmed and deduped case-insensitively so duplicate-looking labels
  cannot produce names such as `SHINY SHINY Part`; the paste-ready Streamer.bot redeem source
  mirrors the same guard.
- **`RedemptionEngine`** (Phase 4, step 1) — the rest of the redemption pipeline, also
  ported from the Streamer.bot action and shared: `SelectCollection` (weighted pick +
  featured-boost multipliers + event-window gating) and `ApplyRedemption` (inventory
  read-modify-write: owned counts, completion detection + seeding, pull-streak/triple,
  dup-protection counter), wrapping `PullEngine.Roll`. Now the native EventSub path,
  Streamer.bot, and MixItUp share the *whole* pull→apply flow, not just the inner roll.
  Smoke-tested (collection weighting, event gating, new/duplicate/completion/streak).
- **`CommandEngine`** (Phase 4, step 1b) — the chat commands, also shared (ported from the
  catalog-commands, collection, and salvage actions): `Inventory`, `Missing`, `Duplicates`,
  `Balance`, `CollectionDetail`, `Leaderboard`, and the `Salvage` write (consumes duplicate
  copies, credits the wallet, mutates inventory). Read commands return chat lines with the
  same ~440-char segmentation; configurable wording comes via a `CommandContext`. Now the
  *entire* pull→apply→command surface is shared, not just the roll. Smoke-tested per command.
- **Multiple active profiles — foundation** — profiles can now be *live* as a set, separate from
  the editing selection: an `active` flag per profile, `SetProfileActive`/`ReadProfileData` on the
  data store, `activate`/`deactivate` API ops, and a one-time backfill so upgrades keep their old
  single-active behavior. A **command-collision guard** hard-blocks activating (or saving, while live)
  a profile whose chat commands clash with another live profile. Backward-compatible; smoke-tested.
  The active-profiles admin UI and native live routing were added after the first foundation pass.
  Still ahead: per-profile overlay URLs and additional UI/status clarity.
- **Revived smoke-test harness** (broken since 0.5) — now also tests `PullEngine`,
  `RedemptionEngine`, `CommandEngine`, active-profile routing/collisions, `AppwriteOptions`,
  and `TwitchOptions`.

## Fixed: row-addressing desync (was "verified 0")

Re-pushing to a non-empty `profile_data` used to report **"verified 0"**, after which
`--cloud` couldn't find the catalog. Cause: `AppwriteDataStore` addressed every row by a
SHA-256-derived id; when a stored row's real `$id` no longer matched the recomputed hash
(after a tenant swap or an earlier hash formula), `UpsertRow(derivedId)` resolved the
`unique_profile_key` conflict against the *existing* row and updated its id, so the
follow-up `GetRow(derivedId)` 404'd and verification failed.

**Fix:** the store now resolves each row by querying the unique index
`(userId, profileId, dataKey)` (confirmed in the console as `unique_profile_key`), uses the
returned row's real `$id` to `UpdateRow`, and `CreateRow(ID.Unique())` when absent — the
derived id is gone. Also fixed two diagnostics (`--appwrite-profiles`, `--appwrite-backups`)
that hardcoded the `local-dev` tenant instead of `ResolveTenant`, so all cloud paths now
agree on the active Twitch tenant. Robust across re-push and login/logout churn; no schema
change needed. Verify live with `--push-to-appwrite` (expect 6/6) → `--cloud`.
