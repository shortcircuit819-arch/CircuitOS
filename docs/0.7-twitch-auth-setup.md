# 0.7 — Twitch OAuth Setup (Phase 3)

Register a Twitch developer app so CircuitOS can log you in with Twitch and use your
real **Twitch user id** as the Appwrite tenant (replacing the `local-dev` placeholder).
This same app is reused in Phase 4 for channel-point reward management + EventSub, so
the scopes below cover both.

Approach: **direct Twitch OAuth** (no Auth0 in the desktop bridge). You register your
own Twitch app; its client id/secret live only on your machine. Auth0 comes in at the
hosted phase.

## 1. Register the Twitch application

1. Go to **https://dev.twitch.tv/console/apps** and log in with your Twitch account.
2. **Register Your Application.**
   - **Name:** anything, e.g. `CircuitOS Dev` (must be unique across Twitch).
   - **OAuth Redirect URLs:** add exactly **`http://localhost:8765`** — must match the
     value the app listens on (configurable, but this is the default).
   - **Category:** Broadcasting Suite (or Application Integration).
   - **Client Type:** **Confidential** (we use the authorization-code flow with a secret).
3. **Create**, then on the app's page:
   - Copy the **Client ID.**
   - Click **New Secret** → copy the **Client Secret** (shown once).

## 2. Save the credentials locally

Create **`twitch.local.json`** in your CircuitOS **Data folder**
(`C:\Users\nicho\Documents\CircuitOS\Data\twitch.local.json`):

```json
{
  "clientId": "your-twitch-client-id",
  "clientSecret": "your-twitch-client-secret",
  "redirectUri": "http://localhost:8765"
}
```

`twitch.local.json` (and the cached `twitch-tokens.local.json`) are gitignored — the
secret never enters source control. **Don't paste the client secret into chat** either.

## 3. Scopes requested at login

The login requests these scopes (needed for Phase 4; harmless for identity now):

- `channel:read:redemptions` — read channel-point redemptions
- `channel:manage:redemptions` — create/manage the channel-point reward

You'll see them on Twitch's consent screen when you log in.

## What I do once this exists

Tell me the app is registered (just confirm — I don't need the secret). Then:

1. `--twitch-login` opens your browser to Twitch, you authorize, and the app captures
   your identity + tokens (cached in `twitch-tokens.local.json` for reuse).
2. The login result shows your Twitch **display name + user id**.
3. `--cloud` then uses your Twitch user id as the Appwrite tenant instead of `local-dev`,
   so your cloud data is keyed to the real streamer. (A one-time re-key/migration from
   `local-dev` → your Twitch id will move the data you already pushed.)

## Notes

- The client secret lives on your machine because this is your own Twitch app — fine for
  the desktop bridge. At the hosted phase the secret moves server-side (one shared app).
- Token caching is plaintext in a gitignored file for dev; Windows DPAPI encryption is a
  hardening follow-up.
