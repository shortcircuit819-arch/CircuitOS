# 0.7 — Cloud Platform + Twitch Integration: Foundation

> Status: **groundwork / design** (started 2026-06-23). No 0.7 code is shipped yet.
> This document is the architectural starting point for the milestone. It is
> meant to be revised as decisions are made.

## Goal

Move CircuitOS from a single-user local Windows app to a multi-tenant hosted web
platform, without rewriting the domain logic. The 0.5 milestone deliberately
routed all data access through `IDataStore` so that 0.7 could be a **swap, not a
rewrite**. This document checks that promise against the real interface and lays
out the work.

Target shape at the end of 0.7:

| Now (local) | 0.7 (cloud) |
|-------------|-------------|
| .NET WinForms + `HttpListener` on `127.0.0.1` | Hosted web backend |
| JSON files on disk (`LocalFileDataStore`) | Appwrite databases + storage (`AppwriteDataStore`) |
| One local user, switchable "profiles" | Many Twitch users, each owning their profiles |
| Streamer.bot C# action writes a local file | Twitch EventSub webhook (Streamer.bot optional) |
| Admin panel at `localhost:8787` | Admin panel in any browser |
| OBS overlay = local file path | OBS overlay = per-profile cloud URL |
| No auth | Twitch OAuth (via Auth0) |

Chosen stack (decided 2026-06-22, see `HANDOFF.md`): **Appwrite** (database,
storage, functions), **Auth0** (Twitch OAuth), **Discord** (notifications).

## Current data layer — what AppwriteDataStore must replicate

Source of truth: [`tools/runtime/IDataStore.cs`](../tools/runtime/IDataStore.cs)
and [`LocalFileDataStore.cs`](../tools/runtime/LocalFileDataStore.cs).

**Six data keys**, each currently one JSON file per profile:

| Key | File | Notes |
|-----|------|-------|
| `catalog` | `components.json` | collections, parts, tiers, variants |
| `boost` | `featured-boost.json` | featured-stream multipliers |
| `inventory` | `inventory.json` | viewer inventories — the hot, frequently-written file |
| `roles` | `discord-role-awards.json` | completion-award queue |
| `profile` | `system-profile.json` | branding, terminology, messages, colors |
| `overlay-config` | `overlay-config.json` | overlay appearance/labels |

**Write semantics** (`WriteAtomic`): write temp → parse-validate the temp → swap
into place, capturing the previous version as a timestamped backup. Any cloud
implementation must preserve "validate before commit" and "previous version is
recoverable."

**Backups:** only four keys are "managed backups" (catalog, boost, roles,
profile — **not** inventory or overlay-config). Filenames are
`{label}_{yyyyMMdd}_{HHmmss}_{fff}.json`. `inventory.json` is deliberately
excluded from listing/preview/restore.

**Binary blob:** one overlay background image per profile (`bg.{png,jpg,gif,webp}`).

**Profiles:** a folder per profile under `profiles/<id>/`, a `profile-meta.json`
(`id`, `name`, `createdAt`), and an `active-profile` pointer file. Includes a
one-time migration that moves pre-0.5 root-level data into `profiles/default/`.

## Seam analysis — is `IDataStore` cloud-ready?

Not yet. The interface has three categories of members:

### 1. Cloud-portable (the core data ops) ✓

`Exists`, `ReadRequired`, `TryRead`, `WriteAtomic`, `GetInfo`, `ListBackups`,
`FindBackup`, `ReadBackupJson`, `SaveBackground`, `FindBackground`. These map
cleanly onto Appwrite documents + storage. `WriteAtomic`'s validate-then-commit
becomes "validate in the function, then upsert the document."

### 2. Filesystem-leaky (must change) ✗

- **`string DataPath`** and **`string BackupPath`** expose raw local paths.
  They are consumed in two places that are *not really data-store concerns*:
  - **Streamer.bot path injection** — `CircuitService.Core.cs` escapes `DataPath`
    into the generated C# so the action writes `inventory.json`/overlay state to
    the right folder. In the cloud there is no local path; a Streamer.bot action
    (if still supported) must POST to a cloud API endpoint with the streamer's
    token instead of writing a file.
  - **Overlay serving** — `Program.cs` publishes overlay statics into `DataPath`
    and serves them. In the cloud the overlay is a hosted URL per profile.
  - **`SaveBackground`/`FindBackground` return file paths** — must return URLs /
    storage references instead.

  → These path-shaped responsibilities should move out of `IDataStore` into a
  separate seam (e.g. an `IIntegrationEndpoints` / overlay-URL provider), so the
  data store is purely data.

### 3. Multi-tenancy-reshaped ✗

`ListProfiles`, `CreateProfile`, `SwitchProfile`, `DeleteProfile`,
`RenameProfile`, `ImportProfileData`, `ActiveProfileId`. These assume a single
local operator toggling a global "active profile." In the cloud:

- The **authenticated Twitch user is the tenant.** A user may still own several
  profiles (multiple games), so a profile becomes a row owned by `userId`.
- **`SwitchProfile`/`ActiveProfileId`** become per-user UI state, not a global
  pointer. Every data op must be scoped by `(userId, profileId)`.
- **`DeleteProfile`/cross-profile import** must enforce ownership (a user can
  only touch their own profiles) — an authorization concern that does not exist
  locally.

### Also: synchronous vs async

`IDataStore` is fully synchronous. The Appwrite SDK is async. A hosted backend
should be async end-to-end, so the cleanest path is to convert `IDataStore` to
return `Task<...>` and ripple that through the `CircuitService.*` callers.
(Blocking on async inside a sync interface is possible but is the wrong default
for a web backend.)

## Target Appwrite data model

Two viable models. Recommend starting with **A** to honor "swap not rewrite,"
then evolve hot paths toward **B** where server-side queries pay off.

- **A — document-per-key (minimal swap):** one Appwrite collection
  `profile_data` with documents keyed by `(userId, profileId, dataKey)`, each
  storing the JSON blob in a large string attribute. `AppwriteDataStore` becomes
  a near-drop-in: read/write whole documents exactly like files today. Simple,
  fast to ship, preserves all domain logic. Downside: no server-side querying;
  whole-document reads/writes (fine for catalog/profile; less ideal for a large
  `inventory`).
- **B — modeled collections (cloud-native, later):** `components`, `inventory`
  (per-viewer documents), `boost`, etc. as proper Appwrite collections with
  typed attributes. Enables server-side leaderboards, per-viewer inventory
  writes (no whole-file rewrite), and EventSub handlers that touch one row.
  More work; do it for `inventory` first since it is the hot path.

Other mappings:
- **Backups** → either a `config_backups` collection (timestamped JSON docs,
  TTL/retention policy) or rely on Appwrite's managed backups. Keep the
  "previous version recoverable" guarantee.
- **Overlay background + statics** → Appwrite Storage bucket; overlay served from
  a cloud URL embedding the profile id.
- **Auth** → Auth0 brokers Twitch OAuth; the Twitch user id is the tenant key.
  Appwrite documents carry per-user read/write permissions so tenants are
  isolated at the database layer, not just in app code.

## Twitch integration

**Design principle — native Twitch is the one-stop shop, zero-config.** The whole
point of 0.7 is that a streamer logs in with Twitch and is *done*: no pasting C#
into Streamer.bot, no manually creating a channel-point reward. Streamer.bot (and,
later, MixItUp — see 0.8) are *optional alternatives* for users who already run
them, never a requirement. Integration priority:

1. **Native Twitch (primary, zero-config)** ⭐
2. Streamer.bot (optional, for existing users)
3. MixItUp (optional, 0.8)

The native path:

- **CircuitOS manages the channel-point reward via the Helix API.** After Twitch
  OAuth, CircuitOS creates/updates the custom reward on the streamer's channel
  itself (`POST /helix/channel_points/custom_rewards`), so the streamer never sets
  one up by hand. The reward name/cost come from the profile.
- **EventSub** delivers redemptions: a
  `channel.channel_points_custom_reward_redemption.add` subscription webhooks into
  an Appwrite Function, which runs the pull logic and writes inventory — no
  Streamer.bot in the loop.
- The pull/roll/variant/tier logic is now extracted into one shared place:
  **`tools/runtime/PullEngine.cs`** (ported from `StreamerbotReedeem.txt`, pure over
  the catalog JSON, RNG injected for tests). The native EventSub function, the
  Streamer.bot forwarder, and MixItUp all call this so they never drift apart. It is
  covered by a distribution test in the smoke-test harness (tier weighting, variant
  rate, dup protection, equal-odds fallback). Not yet wired into a live path — that
  happens when the EventSub function is built (Phase 4).

The optional paths reach the *same* shared pull logic:

- **Streamer.bot** — its action POSTs to a cloud endpoint with the streamer's token
  instead of writing a local file (still no per-pull C# logic to maintain in the
  action; it just forwards the redemption).
- **MixItUp (0.8)** — same forwarding shape via its webhook/action support.

This requires Twitch scopes for managing channel-point rewards and reading
redemptions (`channel:manage:redemptions` / `channel:read:redemptions`), requested
during the Auth0-brokered OAuth.

## Phased plan (proposed)

0. **Foundation/design** (this doc) — seam analysis, model choice. ← we are here
1. **Refactor the seam** *(local, no cloud yet)* — split the filesystem-leaky and
   profile-management responsibilities out of `IDataStore`; make the data ops
   async; keep `LocalFileDataStore` working so 0.6 behavior is unchanged. This is
   the make-or-break step and is doable entirely offline.
2. **Appwrite project + `AppwriteDataStore`** — stand up Appwrite (self-hosted for
   dev), implement model A behind the refactored interface, prove parity with a
   data round-trip test. *(A compiling scaffold already exists at
   `tools/runtime/AppwriteDataStore.cs` — every member throws `NotReady()` with its
   intended mapping in a comment, no Appwrite SDK dependency yet.)*
3. **Auth** — Auth0 + Twitch OAuth; scope every data op by the authenticated user.
4. **EventSub** — port the redemption logic into an Appwrite Function behind the
   Twitch webhook; keep Streamer.bot as an alternative POST path.
5. **Hosted admin panel + cloud overlay URL** — serve the existing vanilla-JS
   admin panel and the overlay from the cloud.

## Open decisions (need the user / accounts)

- **Appwrite hosting for dev:** **Appwrite Cloud** (decided 2026-06-23 — the user's
  machine can't run Docker/virtualization, and Cloud matches the hosted 0.7 end
  state). Self-hosted Docker remains a later option for data control but is not used
  for dev. Setup + model-A schema in
  [`0.7-appwrite-dev-setup.md`](0.7-appwrite-dev-setup.md). Only the endpoint differs;
  the `AppwriteDataStore` code is identical.
- **Accounts:** Appwrite project, Auth0 tenant, and a Twitch developer app /
  EventSub subscription must be created by the user (credential-bearing setup).
- **Data model A vs B per key**, and whether `inventory` goes straight to B.
- **Migration:** how an existing local profile's JSON is uploaded into the cloud
  on a streamer's first login (a one-time import, analogous to the 0.5 migration).

### Decided

- **Parallel, not a hard cutover** (2026-06-23). The local Windows app stays
  supported on `LocalFileDataStore`; the cloud platform uses `AppwriteDataStore`;
  the concrete store is chosen at runtime. The Phase 1 seam refactor must keep
  `LocalFileDataStore` fully working throughout.
